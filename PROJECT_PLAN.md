# Instant Messenger — План на проекта

**Технологии:** React · Spring Boot · PostgreSQL · WebSockets · JWT

**Функционалности:** Потребители · Публични канали · История на съобщенията · Търсене

---

## Цел представянето на домашни

Всеки реализира **работещ прототип** на своя модул. Фиксирани/mock данни са напълно позволени — целта е функционален UI и ясни граници на модула, не цялостен backend.

---

## Задачи

### Човек 1 — Списък с канали и навигация

> Да се реализира прототип на страница, която показва списък с канали и позволява превключване между тях. Съобщенията в тях може да са фиксирани.

- Страница с визуализация на списък с канали
- Превключване между канали (активният канал е маркиран)
- Показване на фиксирани съобщения за всеки канал
- Свързано с `GET /channels` (може да е mock-нато)

---

### Човек 2 — База данни

> Да се реализира прототип на база данни, която ще съдържа информация за потребителите и съобщения.

- Настройка на базата данни (PostgreSQL или MySQL)
- Създаване на таблици: `users`, `channels`, `messages`
- Дефиниране на релации между таблиците
- Repository слой (JPA / Spring Data)
- Начални данни (seed) за тестване

> Не трябва да се отклонява от договорените общи модели по-долу.

---

### Човек 3 — Регистрация и вход

> Да се реализира възможност за регистрация и влизане на потребител, може да е с фиксирани данни за начало.

- Форма за регистрация + endpoint `POST /register`
- Форма за вход + endpoint `POST /login`
- JWT токен, върнат при вход и съхранен client-side
- Може да се използват фиксирани данни за достъп в началото

---

### Човек 4 — Съобщения в реално време

> Да се реализира прототип на изпращане на съобщения в реално време.

- UI за въвеждане на съобщение в активния канал
- WebSocket връзка за изпращане и получаване на съобщения
- Съобщенията се появяват без презареждане на страницата
- Може да се използва mock потребител и фиксиран канал в началото

---

### Човек 5 — Търсене

> Да се реализира прототип на функционалност за търсене в съобщения и визуализация на резултатите. Може да се търси само в един канал и съобщенията да са фиксирани.

- Поле за търсене по ключова дума
- Филтриране на съобщения в рамките на един канал
- Ясна визуализация на резултатите
- Свързано с `GET /search?query=...&channelId=...` (може да е mock-нато)
- Съобщенията може да са фиксирани

---

## Общи договори

**Трябва да бъдат договорени от целия екип преди да започне работата. Не се променят без общо съгласие.**

### Модели на данните

```json
User: { 
  id: number, 
  username: string 
}

Channel:
{ 
  id: number, 
  name: string 
}

Message: 
{ 
  id: number, 
  content: string, 
  userId: number, 
  channelId: number, 
  timestamp: string 
}
```

### API Endpoints

| Метод  | Endpoint                          | Отговорник |
|--------|-----------------------------------|------------|
| GET    | `/channels`                       | Човек 1    |
| GET    | `/messages?channelId=...`         | Човек 4    |
| POST   | `/messages`                       | Човек 4    |
| POST   | `/register`                       | Човек 3    |
| POST   | `/login`                          | Човек 3    |
| GET    | `/search?query=...&channelId=...` | Човек 5    |

## Правила

1. API договорите са **заключени** — без едностранни промени
2. Всички използват **едни и същи общи модели**
3. Всеки модул трябва да е **самостоятелно стартируем** с mock данни
4. Финалната интеграция обединява всички модули в едно общо репо
